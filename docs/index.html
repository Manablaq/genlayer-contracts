Overview

This guide documents real development experience from building four Intelligent Contracts on GenLayer Studio. It covers everything that worked, everything that didn't, and why.

GenLayer's official documentation explains what Intelligent Contracts are. It doesn't tell you what silently fails, what state type doesn't persist, or what will crash your contract at deploy time without a useful error message.

This guide fills that gap. Every bug, every fix, and every pattern documented here was discovered by building real contracts — not by reading docs.

ℹ Who this is for

Developers building Intelligent Contracts on GenLayer Studio who want to avoid the common pitfalls and ship working contracts faster.

Key Concepts

Before diving in, here is a plain-English explanation of every GenLayer-specific term used in this guide.

Intelligent Contract
A smart contract that can fetch live data from the internet and use AI to process it — all on-chain, verified by validators.
Validator
A node on the GenLayer network that independently runs your contract's fetch logic to verify the result.
Consensus
Multiple validators must agree on the result of a transaction before it is stored on-chain.
prompt_comparative
A built-in function that runs your fetch code across all validators and uses AI to check if they all got the same answer.
Equivalence string
The rule you write to tell the AI what counts as "the same" result between two validators. Must match your data's volatility.
GenLayer Studio
The browser-based IDE where you write, deploy, and test Intelligent Contracts on the testnet.
exec_prompt
A built-in function that sends a prompt to an AI model and returns the response as a string.
Finalized
A transaction status meaning all validators reached consensus and the result has been written to the chain.

Working APIs

GenLayer validators need to independently replicate every web fetch to reach consensus. This means only public APIs without authentication can be used.

⚠ Why private APIs don't work

If your API requires OAuth tokens, API keys in headers, or HMAC signatures, validators cannot independently replicate the call — so consensus will always fail. Stick to fully public endpoints.

Verified Public APIs

These three APIs were tested and confirmed working on GenLayer Studio across all four contracts.

Binance Public Ticker VERIFIED
https://api.binance.com/api/v3/ticker/24hr?symbol=BTCUSDT

Returns 24-hour price stats for any trading pair. No API key required. Used in CryptoOracle to fetch BTC, ETH, SOL, BNB, XRP, ADA, DOGE, LINK, MATIC, and AVAX prices.

python
url = f"https://api.binance.com/api/v3/ticker/24hr?symbol={symbol}"
raw = gl.nondet.web.render(url, mode="text")
data = json.loads(raw)
price = float(data["lastPrice"])

Real-time weather data for any coordinates. No API key required. Used in WeatherOracle for 13 cities including Lagos, London, New York, Tokyo, Dubai, and more.

url = (

API Pattern: Always wrap in try/except

Always handle API failures inside the fetch closure. Return a structured error JSON rather than letting an exception crash the validator consensus.

python — correct pattern
def fetch() -> str:
    try:
        raw = gl.nondet.web.render(url, mode="text")
        if not raw or raw.strip() in ["null", "", "null\n"]:
            return json.dumps({"error": "API is unavailable.", "status": "unavailable"})
        # parse and return data
        return json.dumps({"data": ..., "status": "ok"}, sort_keys=True)
    except Exception:
        return json.dumps({"error": "API is unavailable.", "status": "unavailable"})

These are bugs confirmed during development. None of them produce a useful error message, which makes them especially dangerous for new developers.

Bug 01 — web.get() stores nothing

CONFIRMED This is the most dangerous bug because it produces zero errors. Your transaction finalizes, Studio shows "Finalized," and your state stores an empty string every single time.

🚨 Why this is dangerous

You won't know anything is wrong until you call a read method and find your state is empty. The transaction log shows success. There is no error. You will waste hours debugging code that is not broken.

python — wrong ❌
# This looks correct. It compiles. It deploys. It finalizes.
# But response.body is always an empty string on Studio.
response = gl.nondet.web.get(url)
self.store = response.body  # ← always ""
python — correct ✅ (from CryptoOracle)
@gl.public.write
def fetch_price(self, symbol: str) -> typing.Any:
    url = f"https://api.binance.com/api/v3/ticker/24hr?symbol={symbol}"

    def fetch() -> str:
        try:
            raw = gl.nondet.web.render(url, mode="text")
            if not raw or raw.strip() == "null":
                return json.dumps({"error": "API down.", "status": "unavailable"})
            data = json.loads(raw)
            return json.dumps({
                "price": float(data["lastPrice"]),
                "status": "ok"
            }, sort_keys=True)
        except Exception:
            return json.dumps({"error": "API down.", "status": "unavailable"})

    fresh = gl.eq_principle.prompt_comparative(
        fetch,
        "They are equivalent if both show an API error, "
        "or if price values are within 2% and symbol matches."
    )
    all_data = json.loads(self.store)
    all_data[symbol] = json.loads(fresh)
    self.store = json.dumps(all_data, sort_keys=True)
💡 Rule

Always use web.render(mode="text"). Never use web.get() with response.body.

Bug 02 — TreeMap state does not persist on Studio

CONFIRMED TreeMap works perfectly in local testing but loses all data after every transaction when deployed to GenLayer Studio.

python — wrong ❌
class CryptoOracle(gl.Contract):
    prices: TreeMap[str, str]  # State is lost after every transaction on Studio
python — correct ✅ (all 4 contracts use this)
class CryptoOracle(gl.Contract):
    store: str  # JSON string — persists correctly on Studio
    pulse: str  # Freshness / metadata tracking

def __init__(self):
    self.store = "{}"  # Empty JSON object
    self.pulse = "{}"

# Writing state
all_data[symbol] = json.loads(fresh)
self.store = json.dumps(all_data, sort_keys=True)  # sort_keys=True is required

# Reading state
all_data = json.loads(self.store)
entry = all_data.get(symbol)
⚠ Always use sort_keys=True

Without sort_keys=True, Python dict key ordering is non-deterministic. Two validators can produce different JSON strings from identical data, causing consensus to fail.

Bug 03 — sender_address causes deploy failure

CONFIRMED Using gl.message.sender_address inside __init__ causes a runtime error at deploy time. The contract will not deploy. The error message is not helpful.

python — wrong ❌
def __init__(self):
    self.owner = gl.message.sender_address  # Runtime error — contract won't deploy
python — correct ✅ (from MultiKeyVault)
def __init__(self, owner_address: str):
    # Pass your MetaMask wallet address as a parameter when deploying
    assert is_valid_address(owner_address), "Invalid owner address."
    self.owner           = owner_address
    self.allowed_callers = json.dumps([owner_address])

Bug 04 — run_nondet_unsafe fails with web fetching

CONFIRMED Combining run_nondet_unsafe with any web fetch operation fails at runtime. The error message does not indicate the actual cause.

🚨 Fix

Use gl.eq_principle.prompt_comparative for all operations that fetch web data. Do not use run_nondet_unsafe with web fetches.

Bug 05 — gl.vm.UserError in __init__ prevents deploy

CONFIRMED Raising gl.vm.UserError inside __init__ causes the contract deploy to fail entirely.

python — wrong ❌
def __init__(self, rate_limit: int):
    if rate_limit <= 0:
        raise gl.vm.UserError("Rate limit must be positive")  # Breaks deploy
python — correct ✅
def __init__(self, rate_limit: str):
    assert rate_limit.isdigit() and int(rate_limit) >= 1, "Rate limit must be a positive number."
    self.rate_limit = rate_limit  # Note: stored as str, not int
💡 Rule

Use assert in __init__. gl.vm.UserError is safe to use in write methods after deployment.

Best Practices

These patterns were discovered through real development. Each one solved a real problem across the four contracts.

Calibrate your equivalence string

When your contract fetches data, multiple validators independently run the same fetch and compare results. The equivalence string is the rule that tells the AI whether two validators got "the same" answer.

It must match how volatile your data actually is — a price that changes every second needs a different rule than a weather temperature that changes every hour.

Contract & DataEquivalence RuleWhy
CryptoOracle — live price"within 2% and symbol matches"Prices move between validator calls
WeatherOracle — current"within 1°C, 5% humidity, 3 km/h wind"Sensor precision tolerance
WeatherOracle — forecast"within 2°C, dates match exactly"Less real-time, wider tolerance
SocialOracle — trending"at least 3 matching story titles"Stable feed, high overlap expected
SocialOracle — latest"at least 2 matching titles or IDs"Volatile feed, lower threshold
⚠ Get this wrong and consensus breaks

Too strict → consensus fails on valid data. Too loose → wrong data passes. Match the threshold to how much your data can legitimately differ between two validator calls made seconds apart.

State management rules

All state fields in a GenLayer contract must be typed as str. This means booleans, integers, and complex data structures all need to be serialized.

python — from MultiKeyVault
class MultiKeyVault(gl.Contract):
    secrets:         str   # {"name": "value"} — JSON object
    is_paused:       str   # "true" or "false" — NOT a bool type
    rate_limit:      str   # "100" — NOT an int type
    key_version:     str   # "1", "2", "3" — NOT an int type
    audit_log:       str   # JSON array

# Comparing a boolean
assert self.is_paused == "false", "Vault is already paused."

# Incrementing an integer
self.key_version = str(int(self.key_version) + 1)

# Sentinel value — distinguishes "never fetched" from "fetched but empty"
self.pulse = "never"  # Used in SocialOracle

Use module-level functions for shared fetch logic

You can define helper functions outside your contract class and call them from inside closures. This avoids duplicating fetch logic across multiple write methods.

python — from SocialOracle and WeatherOracle
# Defined outside the class — works correctly in closures
def _pull_stories(feed: str, limit: int = 10) -> str:
    try:
        ids_raw = gl.nondet.web.render(
            f"{HN_API}/{feed}stories.json", mode="text"
        )
        ids = json.loads(ids_raw)
        items = []
        for story_id in ids[:limit]:  # Multiple web.render() calls are fine
            raw = gl.nondet.web.render(f"{HN_API}/item/{story_id}.json", mode="text")
            s = json.loads(raw)
            items.append({"title": s.get("title"), "score": s.get("score", 0)})
        return json.dumps({"items": items, "status": "ok"}, sort_keys=True)
    except Exception:
        return json.dumps({"error": "API down.", "status": "unavailable"})

class SocialOracle(gl.Contract):
    @gl.public.write
    def fetch_trending(self) -> typing.Any:
        def fetch() -> str:
            return _pull_stories("top", 10)  # Call the module-level function
        fresh = gl.eq_principle.prompt_comparative(fetch, "...")

Closures in loops — a subtle bug risk

When you define a closure inside a for loop, Python captures the variable by reference, not by value. By the time the closure is called, the loop variable may have already changed to the next value.

This pattern exists in fetch_prices (CryptoOracle) and fetch_multiple (WeatherOracle). It worked on Studio because prompt_comparative executed immediately — but do not rely on this.

python — risky ⚠
for symbol in symbols:
    url = f"https://api.binance.com/...?symbol={symbol}"
    def fetch() -> str:
        return gl.nondet.web.render(url, ...)  # Which url? The last one in the loop.
python — safe ✅
for symbol in symbols:
    url = f"https://api.binance.com/...?symbol={symbol}"
    def fetch(u=url) -> str:  # u=url captures the value at definition time
        return gl.nondet.web.render(u, ...)

Structure AI prompts for specific JSON output

When using exec_prompt for AI analysis, always define an exact JSON schema in your prompt. Free-form responses are hard to parse and hard to write equivalence checks for.

Write your equivalence check to validate only the key decision — not the exact wording. Wording can legitimately differ between validators.

python — from WeatherOracle generate_alert
def analyze() -> str:
    prompt = (
        f"Analyze this weather data for {city} and determine if dangerous:\n\n"
        f"{json.dumps(weather)}\n\n"
        "Respond ONLY with JSON:\n"
        '{"alert_level": "safe" or "caution" or "danger", '
        ' "reason": "one sentence", '  
        ' "recommendation": "one sentence"}'
    )
    return gl.nondet.exec_prompt(prompt)

alert = gl.eq_principle.prompt_comparative(
    analyze,
    "Equivalent if they assign the same alert_level."  # Key decision only
)
data[city]["alert"] = json.loads(alert)

Security

Security patterns discovered while building MultiKeyVault — a multi-secret vault with rate limiting, whitelist control, and emergency wipe.

Access control

Because gl.message.sender_address cannot be used in __init__, ownership must be established via a constructor parameter. Always validate the address format before storing it.

python — from MultiKeyVault
def is_valid_address(address: str) -> bool:
    if not address: return False
    if not address.startswith("0x"): return False
    if len(address) != 42: return False
    valid_chars = "0123456789abcdefABCDEF"
    for char in address[2:]:
        if char not in valid_chars: return False
    return True

def __init__(self, owner_address: str):
    assert is_valid_address(owner_address), "Invalid owner address."
    self.owner           = owner_address
    self.allowed_callers = json.dumps([owner_address])

# Check ownership in every write method
def _is_owner(self, address: str) -> bool:
    return address == self.owner

@gl.public.write
def add_key(self, name: str, value: str, owner_address: str) -> typing.Any:
    assert self._is_owner(owner_address), "Access denied. Only the owner can add keys."

Secret management

MultiKeyVault stores API keys on-chain and exposes them only at fetch time by appending them directly to the URL. Keys are never returned in read methods.

python — safe API key usage
@gl.public.write
def fetch_with_key(self, key_name: str, url: str) -> typing.Any:
    secrets = json.loads(self.secrets)
    secret  = secrets[key_name]  # Retrieved but never returned to caller

    def fetch() -> str:
        try:
            # Key appended to URL — never exposed in response
            raw  = gl.nondet.web.render(url + secret, mode="text")
            data = json.loads(raw)
            return json.dumps({"status": "ok", "response": data})
        except Exception:
            return json.dumps({"error": "API call failed.", "status": "error"})

    fresh = gl.eq_principle.prompt_comparative(fetch, "...")
    self.last_response = fresh  # Response stored, key is not in response
⚠ This only works for URL parameter APIs

This pattern works only for APIs that accept the key as a URL parameter — not APIs requiring the key in request headers.

Rate limiting

MultiKeyVault implements per-key rate limiting at the contract level. This prevents any single key from being overused before the owner can reset the count.

python — rate limit check
counts = json.loads(self.call_counts)
count  = counts.get(key_name, 0)
limit  = int(self.rate_limit)

if count >= limit:
    self.last_response = json.dumps({
        "error":  "Rate limit reached for '" + key_name + "'.",
        "status": "rate_limited"
    })
    return

# Proceed with fetch, then increment count
counts[key_name]     = count + 1
self.call_counts     = json.dumps(counts, sort_keys=True)

Emergency wipe

MultiKeyVault includes an emergency wipe function that clears all secrets and locks the vault in a single transaction. Useful if a key is compromised.

python — emergency wipe
@gl.public.write
def emergency_wipe(self, owner_address: str) -> typing.Any:
    assert self._is_owner(owner_address), "Access denied."
    self.secrets     = "{}"          # Clear all keys
    self.call_counts = "{}"          # Clear all counters
    self.is_paused   = "true"        # Lock vault
    self.key_version = str(int(self.key_version) + 1)
    self._log_call("EMERGENCY_WIPE", "all keys cleared and vault locked")

Audit logging

MultiKeyVault logs every significant action with the action type and detail. The log is capped at 50 entries to manage state size.

python — audit log pattern
def _log_call(self, action: str, detail: str) -> None:
    log = json.loads(self.audit_log)
    log.append({"action": action, "detail": detail})
    if len(log) > 50:
        log = log[-50:]  # Keep only the latest 50 entries
    self.audit_log = json.dumps(log)

Studio Tips

Practical tips for working with GenLayer Studio based on real experience deploying and testing all four contracts.

Empty methods panel = schema error

If your contract deploys but the methods panel is empty, there is a schema error in your contract — usually a type annotation issue. Check all method signatures and state field declarations carefully.

💡 Debug tip

Deploy with a minimal __init__ first (just pass). If the methods panel loads, your schema is valid. Then add logic incrementally.

Errors all look the same

Studio does not distinguish between schema errors and runtime errors — both produce similar-looking failure states. Use this to tell them apart:

SymptomLikely Cause
Error appears before any executionSchema error (type annotation issue)
Empty methods panel after deploySchema error in contract definition
Error appears mid-executionRuntime error
Transaction starts then failsRuntime error or gl.vm.UserError

Write methods take 30–90 seconds

Write methods go through validator consensus before finalizing. This is normal. Always wait for the "Finalized" status before calling read methods to check the result.

Test in this order

  1. Deploy the contract
  2. Verify the methods panel shows all your methods
  3. Call a simple read method to confirm the contract is responsive
  4. Call the write method that populates state
  5. Wait for finalization (30–90 seconds)
  6. Call read methods to verify stored data

Leader rotation is normal

You may see more than 5 validators appearing in the transaction log. This is expected behavior — it reflects cumulative leaders across multiple consensus rounds, not the validator set size. Do not debug this.

Private APIs will always fail consensus

If your API requires authentication (OAuth, API keys in headers, HMAC signing), validators cannot independently replicate the call, so consensus will fail. Use only fully public endpoints.
